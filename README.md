# SPI Protocol Design and Verification using SystemVerilog

## Overview

This project implements and verifies the **Serial Peripheral Interface (SPI)** protocol using **SystemVerilog**. The repository includes the RTL implementation of an SPI Master and SPI Slave along with a complete object-oriented verification environment.

The verification environment validates the functionality of the SPI communication by generating randomized transactions, driving them into the DUT, monitoring the output, and comparing transmitted and received data using a scoreboard.

---

## Project Objectives

- Design an SPI Master capable of transmitting 12-bit data.
- Design an SPI Slave capable of receiving serial data.
- Develop a reusable SystemVerilog verification environment.
- Verify correct data transfer between Master and Slave.
- Demonstrate transaction-level verification concepts using mailboxes and object-oriented programming.

---

## SPI Protocol

SPI (Serial Peripheral Interface) is a synchronous serial communication protocol commonly used for communication between microcontrollers and peripheral devices.

### Signals Used

| Signal | Description |
|---------|-------------|
| SCLK | Serial Clock generated by Master |
| MOSI | Master Out Slave In |
| MISO | Master In Slave Out (Not implemented) |
| CS | Chip Select (Active Low) |

### Data Transfer

- Data Width: **12 bits**
- Transmission Order: **LSB First**
- Communication: **Master → Slave**
- Clock Generated by Master

## RTL Modules

### SPI Master

Responsibilities:

- Generates SPI clock (SCLK)
- Controls Chip Select (CS)
- Serializes 12-bit parallel input
- Transmits data on MOSI
- Implements a simple FSM

States:

- IDLE
- SEND

---

### SPI Slave

Responsibilities:

- Detects active Chip Select
- Samples incoming MOSI data
- Reconstructs 12-bit parallel data
- Indicates transfer completion using `done`

States:

- Detect Start
- Read Data

## Verification Components

### Transaction

Represents one SPI transaction.

Contains:

- Input data
- Output data
- new transaction signal

---

### Generator

- Randomizes input transactions
- Sends transactions to Driver
- Synchronizes with Scoreboard

---

### Driver

- Drives randomized data into DUT
- Controls reset
- Initiates SPI transfer
- Sends expected data to Scoreboard

---

### Monitor

- Observes DUT outputs
- Captures received data
- Sends observed data to Scoreboard

---

### Scoreboard

- Compares transmitted and received data
- Reports PASS/FAIL
- Synchronizes Generator for next transaction

---

### Environment

Instantiates and connects:

- Generator
- Driver
- Monitor
- Scoreboard

Controls:

- Reset
- Test execution
- Simulation completion

## Simulation Flow

1. Apply Reset
2. Generator creates randomized SPI transaction.
3. Driver drives transaction to SPI Master.
4. SPI Master serializes and transmits data.
5. SPI Slave receives serial data.
6. Monitor captures received data.
7. Scoreboard compares expected and actual data.
8. PASS/FAIL is reported.
9. Repeat for all randomized transactions.

## Current Limitations

- Supports only **Master-to-Slave** communication.
- MISO line is not implemented.
- Fixed 12-bit data width.
- Single slave supported.
- SPI operating modes (CPOL/CPHA) are not implemented.
- No functional coverage or assertions.
